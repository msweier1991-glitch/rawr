# Implementation Summary: AI-Powered Marketing Pipeline

## Overview

This document summarizes the implementation of a comprehensive AI-powered marketing pipeline for `rawr_`. The pipeline automates SEO, GEO optimization, and content marketing across 9 channels with real backend logic, database persistence, and mandatory approval gates.

## Key Features

### 1. Real AI Generation
- **Claude API Integration**: All content generated by Anthropic's Claude API
- **Structured Prompts**: Each agent uses specific prompts for targeted output
- **Error Handling**: Graceful failure handling with real error states

### 2. Database-Persistent State
- **Supabase Backend**: PostgreSQL database with real-time subscriptions
- **Schema Design**: Four core tables for pipeline management
- **Data Integrity**: Foreign key constraints and data validation

### 3. Mandatory Approval Workflow
- **Critical Gate**: No content publishes without human approval
- **Approval Dashboard**: Real-time approval interface
- **State Transitions**: Clear status flow (pending_approval → approved/rejected → published)

### 4. Multi-Channel Distribution
- **9 Channel Agents**: SEO, Blog, Reddit, X, LinkedIn, Hacker News, Outreach, Podcast, Influencer, UGC Video, Email, PR
- **Stubbed APIs**: Ready for real social media integration
- **Distribution Runner**: Processes approved content for publishing

### 5. Real-Time Frontend
- **Terminal UI**: Live pipeline logs and status
- **Chatbot Assistant**: AI-powered help with marketing tasks
- **Supabase Subscriptions**: Real-time data updates

## Technical Architecture

### Backend/API Layer
- **Serverless Functions**: Vercel API routes for all endpoints
- **Agent Functions**: Each agent as a separate backend function
- **Database Integration**: Supabase client for all data operations
- **Error Handling**: Comprehensive error handling and logging

### Database Schema
```sql
-- pipeline_runs: Main pipeline execution tracking
CREATE TABLE pipeline_runs (
  id UUID PRIMARY KEY,
  company_id UUID NOT NULL,
  started_at TIMESTAMPTZ NOT NULL,
  status VARCHAR(20) NOT NULL CHECK (status IN ('running', 'complete', 'failed')),
  stage VARCHAR(50) NOT NULL
);

-- agent_outputs: Generated content from all agents
CREATE TABLE agent_outputs (
  id UUID PRIMARY KEY,
  run_id UUID NOT NULL REFERENCES pipeline_runs(id),
  agent_name VARCHAR(50) NOT NULL,
  content JSONB NOT NULL,
  status VARCHAR(20) NOT NULL CHECK (status IN ('draft', 'pending_approval', 'approved', 'rejected', 'published')),
  created_at TIMESTAMPTZ NOT NULL
);

-- approvals: Approval records for agent outputs
CREATE TABLE approvals (
  id UUID PRIMARY KEY,
  agent_output_id UUID NOT NULL REFERENCES agent_outputs(id),
  approved_by TEXT NOT NULL,
  approved_at TIMESTAMPTZ NOT NULL,
  decision VARCHAR(10) NOT NULL CHECK (decision IN ('approved', 'rejected'))
);

-- pipeline_events: Event logging for terminal UI
CREATE TABLE pipeline_events (
  id UUID PRIMARY KEY,
  run_id UUID NOT NULL REFERENCES pipeline_runs(id),
  message TEXT NOT NULL,
  created_at TIMESTAMPTZ NOT NULL
);

-- company_context: Company-specific data for agents
CREATE TABLE company_context (
  id UUID PRIMARY KEY,
  company_id UUID NOT NULL,
  name TEXT,
  description TEXT,
  brand_voice TEXT,
  past_content JSONB,
  competitors JSONB,
  target_keywords JSONB,
  created_at TIMESTAMPTZ NOT NULL,
  updated_at TIMESTAMPTZ NOT NULL
);
```

## Implementation Details

### 1. Agent Implementation
Each agent follows the same pattern:
1. **Fetch Context**: Retrieve company data from database
2. **Build Prompt**: Create specific prompt for the agent
3. **Call LLM**: Use Claude API to generate content
4. **Save Output**: Store in `agent_outputs` with `pending_approval` status
5. **Log Event**: Record completion in `pipeline_events`

### 2. Approval Workflow
1. **Dashboard**: List all `agent_outputs` with `pending_approval` status
2. **Expand Content**: Show full generated content
3. **Approve/Reject**: Call `/api/approvals` endpoint
4. **Update Status**: Change output status based on decision
5. **Distribution**: Approved content moves to distribution queue

### 3. Distribution
1. **Queue Processing**: Run distribution for approved content
2. **Platform-Specific**: Call appropriate distribution function
3. **Update Status**: Mark outputs as `published`
4. **Log Events**: Record distribution attempts

### 4. Frontend Integration
- **Terminal Log**: Reads from `pipeline_events` table
- **Agents Feed**: Reads from `agent_outputs` table
- **Chatbot**: AI-powered assistant with marketing help
- **Documents**: Company context and strategy documents
- **Competitors**: Real-time competitor analysis
- **Workflows**: Automated workflow management
